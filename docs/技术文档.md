## 1. 项目背景

本项目为本人**毕业设计**作品，目标用户为在校师生。校园日常存在信息分散、获取成本高的问题：

- **课程信息**分散在学校教务系统（EAMS），师生需手动登录查询，且缺少「上课前提醒」；
- **天气、宿舍电量**等生活信息分散在不同系统或需现场查看；
- **重要通知**缺乏统一、主动的触达渠道。

为此，项目构建了一套「采集 → 聚合 → 主动推送」的校园信息中台：

1. 通过爬虫自动从教务系统获取课表，按周次推算每日课程并支持课前提醒；
2. 对接第三方天气服务（和风天气）采集实时/逐时天气与预警；
3. 采集宿舍电量数据，支持余额查询与低电量提醒；
4. 通过**企业微信机器人 Webhook** 将课程、天气、电量、自定义通知统一推送到群组；
5. 提供 React 管理后台，集中展示与配置上述信息，并内置完善的账号安全体系（MFA、IP 黑名单、会话管理等）。

项目当前**已部署上线**，运行于 Ubuntu 服务器。

## 3. 系统架构设计

### 3.1 技术栈总览

| 分层 | 技术选型 |
|---|---|
| 表现层（前端） | React 19、TypeScript、Vite 5、Ant Design 5、axios、react-router v7、ECharts |
| 接口层（后端） | Flask 3.1、Flask-CORS、Flask-Limiter（限流）、自研 JWT 认证中间件 |
| 业务层 | 蓝图（Blueprint）模块化路由、Service/Repository 分层、APScheduler 定时任务 |
| 采集层 | Playwright（Chromium 无头）爬取教务系统；requests 对接和风天气 API |

### 3.3 部署架构要点

- **前后端分离**：前端 `npm run build` 产出 `dist/`，由 Nginx 直接托管静态资源；后端仅暴露 `/api` 接口。
- **后端入口**：`run.py` → `create_app()`，读取 `.env` 的 `HOST`/`PORT`/`DEBUG`。生产环境 `DEBUG=false`。
- **爬虫隔离**：课程/电量爬虫以**独立子进程**运行（非 Web 工作线程），避免阻塞 API 请求；进程状态写入 `task_processes` 表，前端可实时查看进度。
- **推送通道**：消息统一经企业微信机器人 Webhook 发出，后端不持有用户手机号等 PII，仅维护 Webhook URL。

### 3.4 关键设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| Web 框架 | Flask | 项目早期选定，轻量、生态成熟；路由按蓝图拆分清晰 |
| 认证 | 自研 JWT 双令牌 + 服务端会话 | Access（1h）用于鉴权，Refresh（7d）用于续期；服务端 `server_sessions` 支持主动撤销 |
| 强认证 | 强制 MFA（TOTP） | 管理后台涉及推送配置与账号管理，开启二次验证降低盗号风险 |
| 采集方式 | Playwright 无头浏览器 | 教务系统含动态渲染与验证码，需真实浏览器环境 |
| 限流 | Flask-Limiter | 防止接口被刷、暴力破解登录 |
| 配置中心 | `module_configs` 表 + `.env` | 运行时可改配置入库，敏感项不回显 |

### 4.2 模块详述

#### M1 课程信息自动获取
- **登录（CAS 单点 + 验证码 OCR）**：`app/cqie-course-timetable/main.py` 用 Playwright 打开学校 CAS 登录页（`cas.cqie.cn/cas`），下载验证码图片后经 `pytesseract` OCR 识别（要求纯数字、长度 ≥4，否则刷新重试，最多 3 次），填充账号/密码/验证码并原生点击登录；登录成功后随机缓冲 3~7 秒，规避「刚进就被限流掐断」。
- **反限流注入**：注入脚本压制 layui/easyui 的「点击过快/请勿频繁」弹窗（仅移除视觉警告 DOM，不阻断 `searchTable` 提交与渲染，避免退避冲掉已渲染数据）。
- **抓取（两种路径）**：默认仅抓服务端渲染的**当前周**；带 `--all-weeks` 时选择「全部教学周」，从页面 JS 的 `TaskActivity` 中提取每门课的**整学期周次位图**（无需逐周切换），并轮询至活动数量稳定再解析。
- **周次权威源（xlsx）**：eams 网页「全部」视图存在漏渲染个别课程实例的问题，因此以服务端导出的 `课表.xlsx` 作为整学期课表的**权威数据源**（`xlsx_import.py` 解析并把 `[2-5] [7-11单] [12-18]` 类周次标注展开为整数周次列表）。
- **处理与入库**：`course_processing/` 将原始数据清洗为结构化记录，`pipeline.save_to_database()` 合并式写入 `courses` 表（按学期子目录隔离）。
- **周次推算**：`get_week_number` 基于 `course_weeks`（第 1 周起始日）动态推算「当前是第几周」，确保课前提醒准确（**已修复**早期用 `MAX(week_number)` 误判的问题）。
- **子进程隔离**：爬虫始终经 `spider_runner.run_spider_process()` 以独立子进程运行（统一注入 `JWXT_HEADLESS`、Tesseract 路径、超时与 UTF-8 编码），不占用 Web 工作线程；进度写 `task_processes` 表供前端轮询。
- **推送**：规则引擎在每节课前触发，经企业微信 Webhook 推送「X 分钟后上 Y 课 @Z 教室」。

#### M2 天气查询
- 对接**和风天气 API**（Ed25519 JWT 认证），采集实时天气、逐时预报、气象预警。
- 数据落 `weather_records` / `weather_alerts`；预警去重（`is_pushed`）后推送，避免重复打扰。

#### M3 宿舍电量查询
- 通过采集脚本获取宿舍电量，落 `electricity_records` / `electricity_remaining` / `electricity_total_capacity`。
- 支持余额查询与「低电量」阈值提醒；Cookie 失效可后台更新。

#### M4 企业微信机器人消息推送
- `webhooks` 表维护多个企业微信群机器人 URL（如「班级群」「测试群」），按 `modules` 字段分别订阅课程/天气/电量/系统消息。
- `custom_pushes` 支持文本/图片/模板消息，支持立即、定时、周期（cron）三种推送类型。
- `push_task_queue` 提供幂等任务队列（task_id 去重）与重试。

#### M5 校园信息聚合展示
- Dashboard 聚合课程、天气、电量、待办推送，配合 ECharts 可视化（电量趋势、天气等）。

#### M6 账号安全（重点）
- **登录**：用户名/密码 →（若启用 MFA）TOTP 二次验证 → 签发 JWT 双令牌 + 登记服务端会话。
- **记住我**：勾选 = 长会话（服务端 30 天 + JWT 闲置 7 天/绝对 30 天）；不勾 = 短会话（服务端 24h + JWT 闲置 2h/绝对 1 天）。
- **会话管理**：`SessionManager` 可查看/强制注销活跃会话；单会话策略（同账号新登录踢旧）。
- **IP 黑名单**：自动/手动封禁异常 IP，`ip_blacklist` 支持过期时间。
- **头像安全**：仅接受 JPG/PNG/GIF/WEBP、校验文件头（Magic Bytes）、限制 2MB、**每自然年最多修改 3 次**。

#### M7 用户与权限
- 用户 CRUD、重置密码、重置 MFA、角色（`role`）、主管理员标记（`is_primary`）。
- `AccessControl` 页面提供访问控制配置。

#### M8 运维中心
- 进程管理（`task_processes`）：查看爬虫/采集进程、进度、可停止。
- 任务中心（`push_task_queue` / 定时规则）：调度与监控。
- 配置中心（`module_configs`）：运行时修改模块参数。

## 5. 数据库设计

### 5.1 选型与连接
- 数据库：**MySQL 8**，字符集 `utf8mb4`。
- 连接串：`mysql+pymysql://<user>:<password>@<host>:<port>/<db>?charset=utf8mb4`（见 `config.py`）。
- ORM：SQLAlchemy 2.0，表结构由模型类定义；`init_db.py` 基于 `create_all` 自动建表/补列（含启动期 `ensure_user_columns()` / `ensure_session_columns()` 自愈补列）。

### 5.2 表清单（共 20 张表）

## 6. 核心业务流程

### 6.1 登录与认证流程
```
浏览器 前端(React) 后端(/api/auth) 数据层
 │ │ │ │
 │ 输入账号密码 │ │ │
 ├─────────────────────►│ POST /login │ │
 │ ├────────────────────────►│ │
 │ │ │ 校验密码(bcrypt) │
 │ │ ├─────────────────────►│
 │ │ │◄─────────────────────┤
 │ │ │ 若启用MFA: │
 │ │ │ 返回 mfa_required │
 │ │◄────────────────────────┤ │
 │ 输入 TOTP 验证码 │ │ │
 ├─────────────────────►│ POST /login/mfa │ │
 │ ├────────────────────────►│ 校验 TOTP │
 │ │ │ 生成 JWT 双令牌 │
 │ │ │ 登记 server_sessions │
 │ │ │ (remember_me 决定时长) │
 │ Set-Cookie: │ │ │
 │ access/refresh/ │ │ │
 │ session_id (httpOnly)│◄────────────────────────┤ │
 │◄─────────────────────┤ │ │
 │ 后续请求带 access │ │ │
 │ access 过期→前端静默 │ POST /refresh │ │
 │ 用 refresh 换发 ├────────────────────────►│ 校验闲置/绝对上限 │
 │ │◄────────────────────────┤ 返回新令牌 │
```
- **令牌有效期**：access 1h；refresh 7d；refresh 受「闲置超时（短 2h / 长 7d）」与「绝对上限（短 1d / 长 30d）」闸门约束。
- **登出**：`/logout` 将当前 jti 加入 `token_blacklist` 并撤销 `server_sessions`。

### 6.2 课程爬取流程
```
管理员触发(立即/预约)
 │
 ▼
scheduled_crawl_tasks 写入任务
 │
 ▼
spider_runner 启动独立子进程 → main.py (Playwright 无头)
 │ 登录 EAMS → 解析课表 → course_table.json
 ▼
course_processing 清洗 → 写入 courses (合并式)
 │
 ▼
重建 course_weeks 周历 (以 week1.start_date 为基准)
 │
 ▼
task_processes 记录进度/完成; 前端实时轮询
```
- **全量 vs 当前周**：`--all-weeks` 走整学期，否则仅抓当前周。
- **无头模式**：由 `.env` 的 `JWXT_HEADLESS` 控制（当前 `true`）。

### 6.3 消息推送流程
```
APScheduler 定时 / 触发
 │
 ▼
构建推送内容 (模板渲染: 课程提醒/天气预警/电量)
 │
 ▼
查询订阅该模块的 webhooks (is_enabled=true)
 │
 ▼
push_task_queue 入队 (task_id 幂等去重)
 │
 ▼
发送企业微信机器人 Webhook (POST)
 │
 ▼
custom_pushes / push_task_queue 更新 status/sent_at
 失败 → retry_count++ 重试
```

> **两条推送链路（实现说明）**：
> - **课表类**（课前提醒、每日/每周课表）走 `rule_service` → `task_service`（队列 + `push_task_queue` 幂等落库）→ `delivery_service`（10s 轮询出队执行，写 `task_processes` 历史）→ `adapter_service` 分发；
> - **天气/电量类**由各自模块任务直接调用 `adapter_service.get_adapter('weather'/'electricity').send(markdown)` 发送，不经队列；
> - Webhook 支持多个（`webhooks` 表 + `.env` 回退），`MultiWeComAdapter` 遍历推送，「至少一个成功即整体成功」；课表图片走 `msgtype=image`（base64 + md5），其余为 `msgtype=markdown`。

### 6.4 采集调度流程（APScheduler）

调度器为 `BackgroundScheduler`（时区 `Asia/Shanghai`），在 master 进程启动一次（Gunicorn 下靠 `preload_app` + `GUNICORN_WORKER` 环境变量避免每个 worker 重复启动）。天气任务仅当配置了和风天气凭证时注册，电量任务仅当配置了电表 Cookie 时注册。核心作业清单：

| Job ID | 触发规则（默认，可后台改） | 说明 |
|---|---|---|
| `spider_job` | cron `0 7,13 * * *` | 日常课表同步（仅当前周，不带 --all-weeks） |
| `rule_check_job` | 每 60s | 检查课表推送规则，生成待推任务 |
| `crawl_dispatch_job` | 每 30s | 扫描预约爬取任务（立即/到期）并起线程执行 |
| `weekly_course_job` | 周一 0:00 | 生成并推送本周课表图片 |
| `clean_processes_job` | 每天 2:00 | 清理 1 个月前进程记录 |
| `clean_sessions_job` | 每天 3:00 | 清理过期服务端会话 |
| `weather_update_now` / `_hourly` / `_alert` | 每 30min / 60min / 10min | 实时/逐时/预警更新（有新预警即推） |
| `weather_daily_report` | cron 07:00 | 每日天气晨报 |
| `weather_analyze_push` | 每 30min | 条件分析推送（雨/高温/低温/预警，带冷却期 + 夜间免打扰 23:00–07:00 + 每日上限 8 条） |
| `electricity_daily` / `_weekly` / `_monthly` | 00:30 / 周一 / 每月 1 日 | 用电日/周/月报（周月附图表） |
| `electricity_cookie_check` | cron 20:00 | Cookie 失效检测 |
| `electricity_low_power` | 每 4h | 低电量检测（≤10 度且距上次提醒 ≥4h 才推） |
| `electricity_full_crawl` | 每周日 03:00 | 电量全量爬取 |

**任务状态自愈**：预约爬取任务若 running/pending 超过 30 分钟（进程被杀等情况）会被 `reap_stale_crawl_tasks()` 自动翻为 `failed`，避免永久卡在「运行中」。所有调度任务状态可在「任务中心 / 进程」页面查看。

## 8. 部署说明

### 8.1 环境要求
- 操作系统：Ubuntu 20.04 / 22.04 LTS（已上线环境）
- Python：3.12+（项目使用 3.12/3.13 验证）
- Node.js：22.x（前端构建）
- MySQL：8.0+
- 浏览器：Playwright Chromium（爬虫用）
- 反向代理：Nginx

### 8.2 后端部署
```bash
# 1. 克隆代码
git clone <repo> Push_System_Flask && cd Push_System_Flask

# 2. 创建虚拟环境并安装依赖
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt
playwright install chromium # 安装无头浏览器

# 3. 配置环境变量
cp .env.example .env # 编辑 .env（见 8.5）

# 4. 初始化数据库（自动建表/补列）
python init_db.py

# 5. 启动（开发/内置）
python run.py

# 生产：必须使用项目自带的 gunicorn_config.py（preload_app=True）
gunicorn -c gunicorn_config.py run:app
```

> **务必用 `-c gunicorn_config.py` 启动，不要用 `gunicorn -w 4 run:app`。**
> 该配置文件设置了 `preload_app=True`，让 Flask app 在 master 进程只加载一次，**APScheduler 调度器只启动一次**。若不加此配置、每个 worker 各自 `create_app()`，则每个 worker 都会启动一份调度器，导致天气/电量等定时任务被执行 N 倍，API 调用次数爆炸并触发限流。配置中还通过 `post_fork` 给 worker 打 `GUNICORN_WORKER=1` 标记，`scheduler.py` 据此二次兜底跳过 worker 启动调度器。
> 默认参数：`bind=127.0.0.1:29528`、`workers=4`（`GUNICORN_WORKERS` 可覆盖）、`timeout=180`（兼顾爬虫长任务）、`max_requests=1000`（防内存泄漏定期重启 worker）。

**systemd 示例（/etc/systemd/system/push-system.service）**
```ini
[Unit]
Description=Campus Push System (Flask)
After=network.target mysql.service

[Service]
User=www-data
WorkingDirectory=/opt/Push_System_Flask
Environment=PYTHONUNBUFFERED=1
ExecStart=/opt/Push_System_Flask/venv/bin/gunicorn -c gunicorn_config.py run:app
Restart=always

[Install]
WantedBy=multi-user.target
```

## 9. 已知问题与优化记录

### 9.1 已知问题 / 限制
1. **爬虫强耦合特定教务系统（EAMS）**：换学校需重写解析逻辑，不具备通用性。
2. **电量 Cookie 易失效**：依赖会话 Cookie，失效需人工更新（`/electricity/cookie`）。
3. **企业微信 Webhook 频率限制**：群机器人有消息频率上限，高并发推送需分批。
4. **无分布式支持**：当前为单实例部署，APScheduler 与进程管理基于本地，未做多节点协调。
5. **前端跨域 cookie**：开发态用跨域地址调后端时，httpOnly cookie 可能存不住，须同源或配置 CORS 凭证。

### 9.2 版本优化记录（v6.8.0 → v6.8.1）
| 版本 | 类型 | 内容 |
|---|---|---|
| v6.8.0 | 功能 | 服务端会话撤销三列（revoked_*）、单会话策略、MFA 强制 |
| v6.8.1 | 修复 | ① 课程周次推算改为基于 `course_weeks.week1.start_date`，**修复「无课却误推送」**；② 登录页新增「记住我」复选框并串接 JWT 闲置/绝对上限（短/长会话）；③ 头像上传安全加固（类型白名单 + Magic Bytes + 2MB + 年改 3 次）；④ 修复会话失效弹框重复出现（仅弹一次）；⑤ 更新隐私/Cookie 政策；⑥ `import magic` 改为可选导入，提升启动兼容性 |
