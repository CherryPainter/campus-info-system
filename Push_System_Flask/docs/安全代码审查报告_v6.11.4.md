## 二、确认安全（关键面，附证据）

### 1. 认证与授权（越权防护）
- `jwt_required`（`auth_middleware.py:65`）校验 Bearer/ cookie 中的 access token，签名算法 HS256 + `SECRET_KEY`；`type != 'access'` 直接 401，refresh/mfa token 不能冒充 API 凭证。
- `admin_required`（`auth_middleware.py:180`）在 `jwt_required` 之上校验 `role == 'admin'`，非管理员返回 403。
- **role 来自 JWT 签名载荷**，普通用户无法自改 role 提权（需 SECRET_KEY 才能签发）——唯一越权前提是 SECRET_KEY 泄露，已在 v6.11.0 改为生产必填、缺失即启动失败。
- 会话管理接口按 role 分层：普通用户只能查/踢自己的会话，越权他人返回 403。
- 结论：**无越权漏洞。**

### 2. SQL 注入
- 业务查询全部走 SQLAlchemy ORM（参数化），如 `session.query(User).filter_by(username=username)`。
- 仅有的几处 `text()` 原生 SQL（`init_db.py:152/168/439`、`session_service.py:470/495`）所用表名/列名/ DDL 均来自**硬编码模型元数据或数据库自省结果**，无任何用户可控输入；字符串默认值已做 `''` 转义。
- 另有 WAF 式 SQL/XSS 检测层（`security.py:scan_request_for_attacks`）对所有 query/form/JSON 扫描，作为纵深防御（即便误报也只拦截，不影响 ORM 正确性）。
- 结论：**无 SQL 注入漏洞。**

### 3. 命令注入
- `spider_runner.py:69` 以**列表形式** `subprocess.run([python, script] + args)`，未使用 `shell=True`；`args` 来自受控调用方（学期 ID 等），无 shell 元字符展开风险。
- `platform_utils.py` 中 `taskkill`/`tasklist` 仅拼接整数 PID（`str(pid)`），无注入面。
- 结论：**无命令注入漏洞。**

### 4. 反序列化
- 全仓库无 `pickle.load(s)` / `yaml.load(未指定Loader)` / `marshal.loads` 等危险反序列化调用（已 grep 确认）。
- 结论：**无反序列化漏洞。**

### 5. 密钥与凭据
- `SECRET_KEY`：生产（`DEBUG=false`）缺失或取值 `dev-secret-key` 直接 `RuntimeError` 启动失败；开发态给不安全默认值并报 `warnings`。
- `ADMIN_TOKEN`：启动校验 ≥16 位。
- 和风天气私钥仅从文件读取（`weather/fetcher.py:33` `open(private_key_path,'rb')`），不落日志。
- 密码 bcrypt 哈希存储（`auth_routes.py`）；初始哈希写入 `data/auth/password.hash`，权限随目录。
- 结论：**密钥管理到位。**

### 6. 其他纵深防御（已具备，确认有效）
- 边缘攻击检测（`security.py:_run_security_checks`）：IP 黑名单、方法白名单、请求体大小（10MB）、敏感路径（.env/.git 等）、路径遍历、SQL/XSS 扫描；命中即记 `ip_security_events` 并可自动封禁。
- 登录信号感知五维滑动窗口 + 限流/限时封禁（登录路径白名单避免误封自己）。
- Flask-Limiter 接入 Redis（未配置降级内存），登录 `strict=10/min`。
- 头像上传校验：类型白名单（拒 SVG）、Magic Bytes、限 2MB、限频。
- 结论：**纵深防御完整。**

## 三、本次修复的漏洞（F1–F4）

### F1 — `.env` 配置注入（中危）
**位置**：`app/api/config_routes.py` `update_config()`
**问题**：接口把用户传入的 `value` 直接 `f.write(f'{k}={v}\n')` 写回 `.env`，且未过滤换行符。管理员（已认证）可在任意**非敏感可编辑**配置项（如 `cors_origins`→`ALLOWED_ORIGINS`、`class_name`→`CLASS_NAME`）的 value 中嵌入 `\n`，注入出 `SECRET_KEY=...`、`DEBUG=true` 等额外行，持久化篡改启动级配置，重启后仍生效（配置投毒 / 越权升级）。
**修复**：
1. 值含 `\r`/`\n` 直接 400 拒绝（`.env` 值本不应含换行）。
2. 增加写入拒绝名单 `_ENV_WRITE_DENYLIST`：`SECRET_KEY / ADMIN_TOKEN / JWT_ADMIN_PASSWORD / DATABASE_* / REDIS_URL / QWEATHER_* / JWXT_PASSWORD / ELECTRICITY_CRAWLER_COOKIE`，即便 DB 侧 `is_sensitive` 标记被改，接口也绝不写入启动级机密。
> 注：`jwxt_password` 在 `module_config.py:159` 已标 `is_sensitive=True`（接口层先于拒绝名单拦截），`ELECTRICITY_CRAWLER_COOKIE` 无对应可编辑配置项，故拒绝名单不破坏任何合法配置编辑功能。

### F2 — 日志注入（低-中危）
**位置**：`app/utils/security.py` `log_request_audit()`（及 `_run_security_checks` 的 UA 采集）
**问题**：审计日志直接记录 `User-Agent`（截取 100 字）但未剥离 CR/LF。任何未认证请求都可构造含 `\r\n` 的 UA，在日志中伪造出额外的"假日志行"，干扰审计与攻击溯源。
**修复**：新增 `_strip_crlf()`，在记录审计与写入 `ip_security_events` 前统一剥离 UA / IP / Path 中的回车换行（`security.py:185` 附近）。

### F3 — 自定义推送图片路径穿越（低危，仅管理员）
**位置**：`app/api/push_routes.py` `create_push()` / `_send_push()`
**问题**：`image_path` 由管理员传入，原逻辑对绝对路径直接采用（`os.path.join(BASE_DIR, p) if not isabs else p`），未限制 `..`。可读取服务器任意文件（如 `.env`、`/etc/passwd`）并经企业微信 webhook 外带。
**修复**：新增 `_resolve_safe_image_path()`，拒绝绝对路径与含 `..` 的段，并用 `os.path.normpath` + BASE_DIR 前缀校验确保最终路径落在项目根内；创建与发送两处均调用，纵深防御。

### F4 — 残留 DEBUG 日志（低危）
**位置**：`app/api/auth_routes.py` `login_mfa()`
**问题**：中存在两行 `logger.info('[DEBUG] 设置 Cookie ...')` / `('[DEBUG] Cookie 设置完成')`，生产环境持续打印且无实质安全价值（与本轮"降低日志噪声"目标相悖）。
**修复**：删除这两行。

## 四、残留风险与建议（信息级，非阻断）

1. **CORS 默认来源含 `localhost:5173/5174`**（`config.py:143`）：开发便利，生产应通过 `ALLOWED_ORIGINS` 环境变量缩到真实前端域名。
2. **`get_client_ip()` 仅在内网段信任 `X-Forwarded-For`**（`security.py:166`）：部署在反向代理后是正确的；若应用直接暴露公网且 `remote_addr` 非内网，则不受 XFF 影响（安全）；若错误地将公网前置机判为"内网"，可能允许伪造 IP 绕过基于 IP 的封禁/限流。请确认生产拓扑。
3. **爬虫 `main.py` 内含 `eval('[' + nameMatch[1] + ']')`**（`main.py:1994`）：这是 Playwright 在**浏览器内**执行的 JavaScript，解析校方页面内容（非用户/攻击者输入），属数据解析手段，非 Python `eval` 注入，风险低；如追求稳妥可改为 `JSON.parse`。
4. **`DATABASE_PASSWORD` 默认 `'123456'`**（`config.py:276`）：仅当 `.env` 缺失该值时生效；当前部署 `.env` 已显式设置，默认值不触发。建议长期改为空并在 `validate_required_config()` 中要求必填（改动涉及启动契约，已记录为建议，未在本轮改动以免破坏开发环境）。
5. **HTTP 环境下 Cookie 非 `secure`**（`auth_routes.py` 依 `FORCE_HTTPS` 决定）：开发态已知行为；生产启用 HTTPS 后自动转为 `secure`。
